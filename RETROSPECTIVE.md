I had an operational need for something like this; I've been building a privacy-based ATProto client (for Bluesky, but not only) which blew up into a whole suite, and I've been doing this largely secretly and alone. I've made a lot of mistakes, but that's part of learning. When thinking of ideas for my portfolio, I wanted something that could be useful even after this portfolio is done; something for long term. Enter horizon-firehose.

The original plan was a Rust-based firehose consumer to potentially replace a Python-based firehose consumer. Assembled a design doc. Directed, while making decisions throughout, following the design doc, and when faced with choices throughout, making decisions which in my view kept the project on the path I wanted and aimed for in the doc.

I'm unfamiliar with the coding language itself; I've only been working in this codebase since Thursday (4/16). What this project composed of primarily was directing Claude Code to convert the design doc into a shippable program. Also, I needed something big to show my work more comprehensively for the Guild, especially with respect to adversarial review and commit history showing over time as I build something following an idea.

The consumer has a sole dependency: Doll's proto-blue Rust SDK, and in the course of developing this, I actually wound up filing four upstream issues and then had a PR merged into proto-blue. That was my first ever open source contribution to an existing repo.

As of now, it's running in my AppView alongside other firehose consumers, and I've opted to run it parallel over cutting over for full replacement, as working with Python requires a small file edit and rebuilding in Docker. When I'm done with that development project, and the frequency of edits in firehose consumers becomes more rare, I will revisit this, because I do feel a bit of pride for having created this, and I like it more. Haha.

When I stop to think about it, there are a couple things I might do differently, and I'll have the chance to in the next project. I aimed for more adversarial reviews (maybe three or four) but chose to leave it at two when the reviewer was finding more reaches than actual issues. This was my first foray into fresh-in-Rust; the two ports of my existing productions from October were an ease-in to the language, but this was the first time I built something in Rust from scratch to shipped. The other thing: Docker involvement should've come earlier in the project rather than at the end. It could've given me more confidence in the code sooner in the process.

I ran two adversarial reviews of the code at different points in the process. Both times I ran them in fresh chats, completely removed from a project folder, and with no context of the scope of the project or any history whatsoever. This helped highlight genuine problems because the instance had no involvement. This was the first project I'd ever run adversarial reviews like this, and this method is much more efficient than previous, where I'd just have Claude attack its ideas in other projects, lol.

There were a lot of moments in the development of this consumer where I didn't quite understand the issue, but it's easy to ask for an explanation and then make an informed decision. Several times Claude mentioned something where I just blanked out mentally; I waited for it to finish its part (or stopped it early to get some clarification) more often than I ever have on a project before. After an explanation in a way I could grasp, I was able to make a better informed decision.

Once it was finally built, and I was installing it on my AppView, it was really exciting watching it build and run, fully functional. Just pure joy.
